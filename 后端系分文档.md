# Lebo 小程序后端系统设计文档

## 1. 项目概述

### 1.1 项目背景
Lebo 是一个微信小程序后端服务系统，提供用户管理、商品管理、订单管理、积分体系、消息通知等功能。采用微信云托管部署，前端通过 callContainer 直接调用后端 API，业务鉴权由后端 JWT 机制统一保障。

### 1.2 技术栈

| 技术 | 版本 | 说明 |
|------|------|------|
| Spring Boot | 3.3.x | 核心框架 |
| JDK | 17（编译目标） | Maven 编译 release=17，容器可用更高 JRE |
| MyBatis-Plus | 3.5.7 | ORM 与分页 |
| MySQL | 8+/9+ | 关系型数据库 |
| Redis | Spring Data Redis | 缓存与扩展能力 |
| WebSocket | Spring WebSocket | 订单状态实时通知 |
| JWT | JJWT 0.12.6 | 身份认证 |
| 微信云托管 | - | 部署平台 |
| 微信云对象存储(COS) | - | 文件存储 |
| Maven | 3.9+ | 构建工具 |

### 1.3 项目结构

```
src/main/java/cn/muzisheng/lebo/
├── api/                    # 控制器层（HTTP 接口）
├── service/                # 业务层接口
│   └── impl/               # 业务实现
├── mapper/                 # MyBatis-Plus 数据访问层
├── entity/                 # 数据库实体映射
├── dto/                    # 入参对象
├── vo/                     # 出参对象
├── model/                  # 通用模型与枚举（Result/Response/状态枚举）
├── config/                 # 应用配置（数据源、Web、WebSocket、存储等）
├── interceptor/            # HTTP / WebSocket 鉴权拦截器
├── handler/                # 类型处理器 / WebSocket 处理器
├── exception/              # 业务异常定义
├── constant/               # 常量定义
├── utils/                  # 工具类
└── LeboApplication.java    # Spring Boot 启动类
```

### 1.4 架构设计要点

1. 分层清晰：`api -> service -> mapper`，控制器不直接操作数据库。
2. 统一鉴权：HTTP 请求统一走 `JwtAuthInterceptor`；WebSocket 握手走 `JwtHandshakeInterceptor`。
3. 统一响应：正常返回 `Result<T>`，异常由 `ExceptionApi` 统一转换为 `ExceptionResponse`。
4. 事务边界：订单支付、订单完结、积分兑换等关键链路使用事务保证一致性。
5. 可观测性：核心路径保留结构化日志，便于追踪订单、积分与消息事件。

### 1.5 文档口径说明（重要）

1. 本文档以当前代码实现为准，重点对齐 `entity`、`enum`、`service/impl` 的业务语义。
2. 若数据库注释与代码语义存在历史差异，以业务实现语义为准，并在文档中显式标注。
3. 金额字段统一按“元”描述（与当前 `schema.sql` 和业务日志口径一致）。

## 2. 数据库设计

### 2.1 用户表 (user)

用户主表，存储用户基本信息。

| 字段名 | 类型 | 说明 |
|--------|------|------|
| open_id | VARCHAR(255) | 主键，微信用户唯一标识 |
| session_key | VARCHAR(255) | 微信会话密钥（加密存储） |
| nick_name | VARCHAR(255) | 用户昵称（唯一） |
| union_id | VARCHAR(255) | 微信开放平台统一标识 |
| password | VARCHAR(255) | 密码（商户后台登录用） |
| avatar | VARCHAR(255) | 头像URL |
| gender | TINYINT | 性别：0-保密，1-男，2-女 |
| city | VARCHAR(255) | 城市 |
| age | INT | 年龄 |
| birthday | VARCHAR(255) | 生日（yyyy-MM-dd格式） |
| email | VARCHAR(255) | 邮箱 |
| phone | VARCHAR(255) | 手机号 |
| status | TINYINT | 状态：0-正常，1-不活跃，2-暂停，3-封禁，4-注销 |
| last_login | DATETIME(6) | 最后登录时间 |
| gmt_created | DATETIME(6) | 创建时间 |
| gmt_modified | DATETIME(6) | 修改时间 |
| is_deleted | TINYINT | 逻辑删除：0-正常，1-删除 |
| is_super | TINYINT | 是否为商户：0-普通用户，1-商户 |

```sql
CREATE TABLE `user` (
    `open_id` varchar(255) NOT NULL COMMENT '微信 openid',
    `session_key` varchar(255) DEFAULT NULL COMMENT '微信会话密钥',
    `nick_name` varchar(255) NOT NULL COMMENT '用户昵称',
    `union_id` varchar(255) DEFAULT NULL COMMENT '微信unionid',
    `password` varchar(255) DEFAULT NULL COMMENT '登录密码',
    `avatar` varchar(255) NOT NULL DEFAULT '' COMMENT '头像地址',
    `gender` tinyint NOT NULL DEFAULT '0' COMMENT '性别 0:保密 1:男 2:女',
    `city` varchar(255) DEFAULT NULL COMMENT '城市',
    `age` int DEFAULT NULL COMMENT '年龄',
    `birthday` varchar(255) DEFAULT NULL COMMENT '生日',
    `email` varchar(255) DEFAULT NULL COMMENT '邮箱',
    `phone` varchar(255) DEFAULT NULL COMMENT '手机号',
    `status` tinyint NOT NULL DEFAULT '0' COMMENT '用户状态 0:正常 1:不活跃 2:暂停 3:封禁 4:注销',
    `last_login` datetime(6) DEFAULT NULL COMMENT '最后登录时间',
    `gmt_created` datetime(6) DEFAULT CURRENT_TIMESTAMP(6) COMMENT '创建时间',
    `gmt_modified` datetime(6) DEFAULT CURRENT_TIMESTAMP(6) ON UPDATE CURRENT_TIMESTAMP(6) COMMENT '更新时间',
    `is_deleted` tinyint NOT NULL DEFAULT '0' COMMENT '逻辑删除 0:未删除 1:已删除',
    `is_super` tinyint NOT NULL DEFAULT '0' COMMENT '是否为商户 0:普通用户 1:商户',
    PRIMARY KEY (`open_id`),
    UNIQUE KEY `uk_nick_name` (`nick_name`),
    KEY `idx_status` (`status`),
    KEY `idx_phone` (`phone`),
    KEY `idx_gmt_created` (`gmt_created`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户信息表';
```

## 4. 核心业务流程

### 4.1 用户登录流程

1. 前端调用微信登录获取 code
2. 后端通过 code 调用微信接口获取 openid 和 session_key
3. 查询或创建用户记录
4. 生成 JWT Token 返回给前端
5. 后续请求通过 Token 鉴权

### 4.2 订单流程

1. 用户创建订单，状态初始化为“待支付”。
2. 用户提交支付，状态进入“已支付”，并通过 WebSocket 通知商户有新订单。
3. 商户处理订单：
   - 接单完成：订单进入“已结束”，执行库存出库、积分结算、积分记录落库、消息通知。
   - 拒单退款：订单进入“已退款”，发送拒单通知。
4. 订单超时未支付由定时任务自动取消。

> 订单字段语义说明：`pay_type` 表示订单状态，`pay_option` 表示支付方式；两者的业务语义以 `OrderTypeEnum` 与 `OrderOptionEnum` 为准。

### 4.3 积分兑换流程

1. 校验商品是否存在、是否可积分兑换
2. 校验商品库存是否充足
3. 校验用户积分是否足够
4. 执行商品出库（创建出入库记录）
5. 扣减用户积分
6. 创建积分记录
7. 发送积分兑换成功消息

### 4.4 数据备份流程

1. 导出：查询所有表数据 → 序列化为 JSON → 压缩为 ZIP → 记录历史操作
2. 导入：校验文件 → 解析 JSON → 清空表数据 → 批量插入 → 记录历史操作

### 4.5 消息中心流程

1. 商户发送消息时，系统按接收人拆分消息记录，并使用 `information_id` 聚合同批消息。
2. 客户拉取消息列表仅返回本人且未删除数据。
3. 客户调用 `/information/look?id={消息主键ID}` 后，消息 `is_look` 从 `0` 更新为 `1`（仅允许更新当前登录用户自己的消息）。

## 6. 配置与运行要点

### 6.1 核心环境变量

| 变量名 | 说明 |
|--------|------|
| `DATASOURCE_URL` | MySQL 连接地址 |
| `DATASOURCE_USERNAME` | MySQL 用户名 |
| `DATASOURCE_PASSWORD` | MySQL 密码 |
| `APP_ID` | 微信小程序 AppID |
| `APP_SECRET` | 微信小程序 AppSecret |
| `WX_CLOUD_ENV` | 微信云托管环境 ID |

### 6.2 鉴权要点

1. HTTP 接口默认走 JWT 拦截器校验。
2. 白名单接口（如登录、注册、文件下载等）在 `WebConfig` 中显式排除。
3. WebSocket 握手时通过 `token` 参数解析用户身份，并注入会话属性。

### 6.3 文档维护建议

1. 涉及数据库字段新增/变更时，同时更新实体、`schema.sql` 和本设计文档。
2. 新增接口时，至少补充“入参、出参、鉴权、业务副作用”四项。
3. 版本发布前执行一次“代码实现 vs 文档字段”差异检查，避免口径偏差。
